### 1.2 设计原则

| 原则 | 实践 |
|------|------|
| **关注点分离** | 四层架构：展示→业务→AI引擎→数据，单向依赖 |
| **异步优先** | 文档解析、向量化、KG构建走 Celery 异步任务 |
| **接口抽象** | 解析器、LLM、Embedding、向量库均定义抽象基类 |
| **配置驱动** | 模型/存储选择通过 YAML 配置文件切换，不硬编码 |
| **优雅降级** | KG 不可用时不影响基础问答；LLM 超时时返回检索结果兜底 |

## 2. 技术选型

### 2.1 后端核心

| 领域 | 技术选型 | 说明 |
|------|---------|------|
| **Web 框架** | FastAPI ≥ 0.110 | 异步原生、自动生成 OpenAPI 文档、性能优异 |
| **ORM** | SQLAlchemy 2.0 (async) | 成熟稳定、支持异步会话 |
| **异步任务** | Celery + Redis | 耗时操作解耦、任务重试机制 |
| **认证** | python-jose (JWT) + bcrypt | 无状态认证、密码安全哈希 |
| **文档解析** | PyMuPDF + python-docx + openpyxl + python-pptx | 各格式公认最优库 |
| **异步 HTTP** | httpx | 调用 LLM API |
| **文本处理** | LangChain (仅 TextSplitter) | 语义分块 |
| **日志** | loguru | 结构化日志 |

### 2.2 AI / 检索核心

| 领域 | 技术选型 | 选型理由 |
|------|---------|---------|
| **LLM** | 通义千问 qwen-max / DeepSeek deepseek-chat | 优先国内API，通过抽象层切换 |
| **Embedding** | text2vec-large-chinese / bge-large-zh-v1.5 | 中文语义理解 Top 级别 |
| **向量数据库** | Milvus (主力) / Qdrant (轻量备选) | 高性能ANN检索、水平扩展 |
| **全文检索** | Elasticsearch 8.x + IK 分词器 | 成熟的 BM25 实现 |
| **图数据库** | Neo4j 5.x Community | 图数据库事实标准、Cypher 查询 |
| **重排序** | bge-reranker-large | Cross-encoder 精排 |

### 2.3 前端核心

| 领域 | 技术选型 | 说明 |
|------|---------|------|
| **框架** | React 18 + TypeScript | 类型安全 |
| **构建** | Vite 5 | 极速 HMR |
| **UI 库** | Ant Design 5 | 企业级组件、中文友好 |
| **状态管理** | Zustand | 轻量、无模板代码 |
| **图谱可视化** | @antv/g6 v5 | 阿里开源、知识图谱场景深度适配 |
| **Markdown** | react-markdown + rehype | 答案渲染 |
| **文件上传** | resumable.js | 大文件分片上传、断点续传 |

### 2.4 基础设施

| 组件 | 技术 | 用途 |
|------|------|------|
| 容器化 | Docker + Docker Compose | 一键部署 |
| 反向代理 | Nginx | SSL 终止、静态文件、负载均衡 |
| 关系数据库 | PostgreSQL 15 | 用户、KB、文档元数据 |
| 缓存/消息 | Redis 7 | 会话缓存 + Celery Broker |
| 对象存储 | MinIO | 原始文件存储、S3 兼容 |

## 3. 核心模块设计

### 3.1 文档解析引擎 (Parser Engine)

采用 **策略模式 + 注册中心**，新增格式只需注册一个新类。

```
                    ┌─────────────────────┐
                    │   ParserRegistry    │
                    │  (按扩展名/MIME路由)  │
                    └─────────┬───────────┘
                              │
        ┌─────────┬───────────┼───────────┬─────────┐
        │         │           │           │         │
   ┌────┴────┐┌───┴───┐┌────┴────┐┌────┴────┐┌───┴───┐
   │PDFParser││DOCX   ││XLSX    ││PPTX    ││Text   │ ...
   │         ││Parser ││Parser  ││Parser  ││Parser │
   └────┬────┘└───┬───┘└────┬────┘└────┬────┘└───┬───┘
        │         │           │           │         │
        └─────────┴───────────┴───────────┴─────────┘
                              │
                    ┌─────────┴───────────┐
                    │  List[Chunk]        │
                    │  (统一输出结构)      │
                    └─────────────────────┘
```

### 3.2 RAG 管道

```
文档 ──→ [分块] ──→ [向量化] ──→ [Milvus存储]

问题 ──→ [向量化] ──→ ┌───────────┐
                      │ 混合检索    │
                      │ 向量+BM25+KG│
                      └─────┬─────┘
                            │
                     [RRF融合排序]
                            │
                     [Cross-encoder精排]
                            │
                     [LLM 生成答案]
```

### 3.3 LLM 网关

```python
class BaseLLM(ABC):
    @abstractmethod
    async def chat(self, messages: List[Message], **kwargs) -> ChatResponse: ...
    @abstractmethod
    async def chat_stream(self, messages: List[Message], **kwargs) -> AsyncIterator[str]: ...

class TongyiLLM(BaseLLM): ...   # 通义千问适配
class DeepSeekLLM(BaseLLM): ... # DeepSeek 适配
```

**故障切换**: 主 LLM 超时/不可用 → 自动切换备用 LLM（配置驱动）。

### 3.4 知识图谱引擎

```
文档块 ──→ [实体抽取: LLM + 规则] ──→ [关系抽取: LLM] ──→ [Neo4j 存储]
```

实体类型：`PERSON`, `ORG`, `DATE`, `MONEY`, `LOCATION`, `TERM`

## 4. 数据流转

### 4.1 文档入库流程

```
用户上传 → FastAPI 接收 → 存入 MinIO
                ↓
         创建 Document 记录 (status=PROCESSING)
                ↓
         Celery 异步任务:
         ├─ [1] 解析引擎 parse → List[Chunk]
         ├─ [2] 文本分块 (语义切分)
         ├─ [3] 批量 Embedding
         ├─ [4] 写入 Milvus + ES
         └─ [5] (可选) KG 实体抽取 → Neo4j
                ↓
         更新 Document 记录 (status=READY, chunk_count=N)
```

### 4.2 问答请求流程

```
用户提问 (含 kb_id)
        ↓
问题 Embedding
        ↓
并行检索 (三路):
├── Milvus 向量检索    → Top-20
├── ES BM25 关键词检索  → Top-20
└── Neo4j 图谱检索      → 关联文档块
        ↓
RRF 融合排序 → Top-10
        ↓
(可选) Cross-encoder 精排 → Top-5
        ↓
构建 Prompt: System + Context(chunks) + Question
        ↓
LLM 流式生成 (SSE) → 前端逐字展示
        ↓
返回: {answer, sources[{doc, page, chunk, score}], confidence}
```

## 6. API 接口设计概要

### 6.1 设计规范

| 规范 | 说明 |
|------|------|
| URL 前缀 | `/api/v1/` |
| 请求格式 | JSON |
| 响应格式 | `{code: int, message: str, data: T}` |
| 认证方式 | `Authorization: Bearer <JWT>` |
| 流式响应 | SSE (text/event-stream) |
| 分页 | `?page=1&page_size=20` |

## 7. 安全架构

### 7.1 认证与鉴权

```
Client → POST /login {user, pass} → bcrypt验证 → JWT Token
Client → API Request (Bearer Token) → JWT Decode + RBAC Check → Response
```

### 7.2 RBAC 角色定义

| 角色 | KB管理 | 文档上传 | 文档删除 | 问答搜索 | 用户管理 | 系统配置 |
|------|--------|---------|---------|---------|---------|---------|
| **admin** | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| **editor** | ✅(自己的) | ✅ | ✅ | ✅ | ❌ | ❌ |
| **viewer** | ❌ | ❌ | ❌ | ✅ | ❌ | ❌ |

### 7.3 数据安全矩阵

| 层面 | 措施 | 阶段 |
|------|------|------|
| 传输安全 | TLS 1.3 + HSTS | P2 |
| 密码存储 | bcrypt + 随机盐 | P0 |
| 认证令牌 | JWT (HS256, 24h过期) | P0 |
| 文档原文 | AES-256-CBC 可选加密 | P2 |
| 审计日志 | 全量记录、不可删除 | P2 |
| API 限流 | Token Bucket (100 req/min per user) | P2 |
