## 2. 硬件需求

| 资源 | 要求 |

## 3. Docker Compose 编排

### 3.1 生产环境 (`app/docker-compose.prod.yml`) ⭐ v2.0 GA

生产环境编排文件位于 `app/docker-compose.prod.yml`，包含 **10 个服务**：

| 服务 | 镜像 | 端口 | 说明 |
|------|------|------|------|
| postgres | postgres:15-alpine | 5432 | 主数据库 |
| redis | redis:7-alpine | 6379 | 缓存 + Celery 消息队列 |
| minio | minio/minio | 9000/9001 | 文件对象存储 |
| milvus-standalone | milvusdb/milvus | 19530 | 向量数据库 |
| elasticsearch | elasticsearch:8.12.0 | 9200 | 全文检索 |
| neo4j | neo4j:5-community | 7474/7687 | 图数据库 |
| backend | 本地构建 | 8000 | FastAPI 应用 |
| celery-worker | 本地构建 | — | 异步任务处理 |
| frontend | 本地构建 | — | React 前端（Nginx 托管） |
| nginx | nginx:alpine | 80 | 反向代理 + 静态资源 |

生产环境特性：
- 所有服务配置 `healthcheck` 健康检查
- `deploy.resources.limits` 资源限制
- `restart: unless-stopped` 自动重启
- 日志驱动 `json-file` + `max-size: 10m`
- 数据库密码通过环境变量注入，非明文
- `depends_on` 控制启动顺序

### 3.2 轻量部署模式 ⭐ v2.0 GA

适用于无需 Docker 的单机场景（Windows/Linux）：

```bash
cd OfficeTool
bash deploy/deploy.sh
```

特点：
- SQLite 替代 PostgreSQL（零配置）
- 同步解析替代 Celery（无需 Redis）
- 函数调用模式 Embedding（本地 sentence-transformers）
- 所有数据存储在本地磁盘

### 3.3 开发环境 (`app/docker-compose.yml`)

用于本地开发调试，各服务暴露端口供 IDE 连接。与生产环境主要区别：
- 端口全部暴露到宿主机
- 不启用资源限制
- ES/Neo4j 密码关闭或使用简单密码

## 4. 部署操作手册

### 4.1 生产环境一键部署

```bash
# 1. 进入 app 目录
cd OfficeTool/app

# 2. 配置环境变量
cp .env.example .env
vim .env  # 填写 LLM API Key、数据库密码等

# 3. 启动所有服务
docker compose -f docker-compose.prod.yml up -d

# 4. 等待服务就绪（约 60-90 秒）
docker compose -f docker-compose.prod.yml ps
# 确认所有服务的 STATE 为 healthy

# 5. 验证
curl http://localhost/api/health
# → {"status": "ok"}

# 6. 浏览器访问
# http://<服务器IP>
# 默认管理员: admin / admin123（首次登录后请立即修改）
```

### 4.2 轻量部署

```bash
cd OfficeTool
bash deploy/deploy.sh
# 按提示完成配置 → 自动启动
```

### 4.3 环境变量清单

```bash
# === 数据库 ===
DATABASE_URL=postgresql+asyncpg://officetool:${PG_PASSWORD}@postgres:5432/officetool

# === LLM API ===
LLM_PROVIDER=deepseek           # deepseek | tongyi
DEEPSEEK_API_KEY=sk-xxx         # DeepSeek API Key
DASHSCOPE_API_KEY=sk-xxx        # 通义千问 API Key (备选)

# === Embedding ===
EMBEDDING_MODEL=shibing624/text2vec-base-chinese

# === 对象存储 ===
MINIO_ENDPOINT=minio:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=<强密码>

# === 安全（生产必改）===
JWT_SECRET_KEY=<openssl rand -hex 32>
DOC_ENCRYPTION_KEY=<openssl rand -hex 32>   # 32 字节，留空则不加密

# === 图数据库 ===
NEO4J_URI=bolt://neo4j:7687
NEO4J_PASSWORD=<强密码>

# === 限流与审计 ===
RATE_LIMIT_ENABLED=true
AUDIT_LOG_ENABLED=true
```

## 5. 配置说明

### 5.1 LLM 配置切换

```yaml
# backend/core/config.py 或 config.yaml

llm:
  provider: tongyi           # 主 LLM
  fallback: deepseek         # 备选（主 LLM 不可用时切换）
  models:
    tongyi:
      chat_model: qwen-max
      api_base: https://dashscope.aliyuncs.com/compatible-mode/v1
    deepseek:
      chat_model: deepseek-chat
      api_base: https://api.deepseek.com/v1
  timeout: 30               # 请求超时 (秒)
  max_retries: 2
```

### 5.2 模型热更新

| 配置 | 影响范围 | 更新方式 |
|------|---------|---------|
| LLM Provider | 问答生成 | 修改配置 → 重启 FastAPI |
| Embedding Model | 新文档索引 | 修改配置 → 需重建向量库 |
| 检索 Top-K | 问答精度 | 修改配置 → 即时生效 |

## 6. 数据备份与恢复

> ⚠️ `scripts/backup.sh` 自动化脚本计划于 v2.1 提供。当前请使用以下手动备份方式。

### 6.1 手动备份

```bash
#!/bin/bash
BACKUP_DIR="/data/backups/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR"

# PostgreSQL
docker compose -f docker-compose.prod.yml exec -T postgres \
  pg_dump -U officetool officetool > "$BACKUP_DIR/db.sql"

# MinIO 文档文件
docker compose -f docker-compose.prod.yml cp minio:/data "$BACKUP_DIR/minio_data"

# Neo4j
docker compose -f docker-compose.prod.yml exec -T neo4j \
  neo4j-admin database dump neo4j --to-path=/backups/
docker compose -f docker-compose.prod.yml cp neo4j:/backups "$BACKUP_DIR/neo4j"

# 打包
tar -czf "$BACKUP_DIR.tar.gz" "$BACKUP_DIR"
echo "备份完成: $BACKUP_DIR.tar.gz"
```

### 6.2 备份策略

| 类型 | 频率 | 保留 |
|------|------|------|
| 全量备份 | 每天 02:00 | 最近 7 天 |
| 增量备份 (PG WAL) | 持续 | 最近 3 天 |
| 异地备份 | 每天 | 最近 30 天 |

### 6.3 恢复流程

```bash
# 1. 停止服务
docker compose down

# 2. 恢复数据库
docker compose up -d postgres
docker compose exec -T postgres psql -U officetool officetool < backup/db.sql

# 3. 恢复文件
docker compose cp backup/minio_data minio:/data

# 4. 恢复 Neo4j
docker compose exec neo4j neo4j-admin load --from=/backups/neo4j.dump

# 5. 重建 Milvus 索引 (从 PG 元数据触发重算)
docker compose up -d
docker compose exec backend python scripts/rebuild_index.py

# 6. 验证
curl http://localhost:8000/api/v1/health
```

## 9. 故障处理

### 9.1 常见故障

| 现象 | 可能原因 | 处理 |
|------|---------|------|
| 问答返回慢 (>30s) | LLM API 拥堵 | 检查 API 状态页；切换备用 LLM |
| 文档上传后一直"解析中" | Celery Worker 挂了 | `docker compose restart worker` |
| 搜索结果空 | Milvus 连接断开 | `docker compose restart milvus` |
| 登录失败 | PG 连接满 | 检查连接数；重启 PG |
| 问答报错 | LLM API Key 过期 | 更新 `.env` 中的 Key → 重启 |
| 前端白屏 | 前端静态文件丢失 | `docker compose up -d --build frontend` |

## 10. 安全加固清单

### 10.1 部署前检查

- [ ] 修改所有默认密码 (PG, Neo4j, MinIO, Redis)
- [ ] 生成随机 `JWT_SECRET_KEY`（`openssl rand -hex 32`）
- [ ] 配置 `DOC_ENCRYPTION_KEY`（生产环境建议开启 AES-256）
- [ ] API 限流启用 (`RATE_LIMIT_ENABLED=true`)
- [ ] 审计日志启用 (`AUDIT_LOG_ENABLED=true`)
- [ ] 防火墙：仅开放 80/443 端口，其他服务端口仅 localhost
- [ ] Docker 镜像固定版本（不使用 `latest` tag）
- [ ] 定期安全更新：`docker compose pull`

### 10.2 v2.0 GA 安全能力

| 能力 | 状态 | 说明 |
|------|:--:|------|
| RBAC 三角色权限 | ✅ | admin / editor / viewer，`@require_role()` 路由保护 |
| 操作审计日志 | ✅ | 全量记录 user/action/target/IP/time/result |
| API 限流 | ✅ | Token Bucket, 100 req/min, 超限 429 |
| AES-256 文档加密 | ✅ | `DOC_ENCRYPTION_KEY` 控制开关 |
| JWT 认证 | ✅ | bcrypt 密码哈希 + JWT token |
| URL Token 认证 | ✅ | 浏览器原生请求（iframe/下载）`?token=` 支持 |

### 10.2 网络安全

- OfficeTool **完全内网部署**，不需要公网 IP
- 仅在调用云端 LLM/Embedding API 时需要**出网访问**
- 出网建议通过企业 HTTP 代理统一出口
- 所有内部服务通信使用 HTTP (内网，无需 HTTPS，除非合规要求)
