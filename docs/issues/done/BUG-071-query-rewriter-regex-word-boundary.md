# BUG-071: 查询改写规则引擎对中英混排文本失效

| 属性 | 值 |
|------|---|
| **严重级别** | 🟡 Medium |
| **影响模块** | 查询改写 → 召回率 |
| **发现方式** | Week 9 运行时测试 |
| **状态** | 🔴 待修复 |
| **发现日期** | 2026-07-11 |

---

## 现象

查询改写规则引擎对中英混排查询（如「微服务和DDD有什么区别」「什么是KPI和OKR」）完全无效，返回空变体列表：

```
输入: "微服务和DDD有什么区别"
期望: ["微服务和领域驱动设计有什么区别"]
实际: []  ← 规则引擎未匹配

输入: "什么是KPI和OKR"
期望: ["什么是关键绩效指标和目标与关键结果"]
实际: []  ← 规则引擎未匹配
```

纯中文查询（如「合同违约金怎么算」）正常工作。

## 根因

`query_rewriter.py:117` 使用 `\b` 正则单词边界匹配英文缩写：

```python
if re.search(r'\b' + re.escape(abbr) + r'\b', query, re.IGNORECASE):
```

`\b` 在 Python re 模块中定义为 `\w` 与 `\W` 之间的边界。中文字符属于 `\W`，英文字母属于 `\w`。在「微服务和DDD」中，「和」与「D」之间是 `\W→\w` 过渡 → 存在 `\b`。但在实际测试中发现 `\bDDD\b` 无法匹配「微服务和DDD有什么区别」。

原因是：中文「和」字符后紧接「DDD」，Python 的 `\b` 在 Unicode 模式下与 ASCII 模式行为不同。`re.IGNORECASE` 不影响 `\b` 行为，但中文字符的 Unicode 类别可能导致 `\b` 判定不一致。

## 影响

- 🟡 含英文缩写的中文查询无法享受规则改写（如 DDD、KPI、OKR、API、NLP 等）
- 🟡 LLM 改写路径正常，但规则引擎本应提供快速、零成本的变体生成
- 🟡 召回率在英文缩写场景下无规则引擎提升

## 修复建议

将 `\b` 边界匹配改为兼容中文字符的正则：

```python
# 方案: 使用前瞻/后顾替代 \b，匹配 ASCII 单词 + 中文字符边界
pattern = r'(?<=[\s一-鿿])' + re.escape(abbr) + r'(?=[\s一-鿿])'
# 同时对字符串开头/结尾也做处理
pattern = r'(?:^|(?<=[\s一-鿿]))' + re.escape(abbr) + r'(?:$|(?=[\s一-鿿]))'
if re.search(pattern, query, re.IGNORECASE):
```
