# BUG-025: KG 实体抽取对短文档返回 0 实体

| 属性 | 值 |
|------|---|
| **严重级别** | 🟢 Low |
| **影响 AC** | Week 6: 实体识别准确率 |
| **发现方式** | 运行时测试 |
| **状态** | Open |

## 现象
Celery 日志显示：
```
实体抽取完成: doc_id=xxx, entities=0, normalized=0
KG 构建完成: doc_id=xxx, entities=0, relations=0
```
上传 346 字符 Markdown 文档后，`build_knowledge_graph` 任务成功执行（Neo4j 连接正常、约束创建正常），但 LLM 实体抽取返回 0 个实体。

## 根因
短文档文本量不足，LLM (DeepSeek) 没有从上下文中找到符合实体定义的模式。这属于 LLM 行为限制，不是代码 bug。实体抽取的 Prompt 设计需要一定长度的上下文才能有效识别实体。

## 影响
- 短文档（< 500 字符）在 Neo4j 中无实体节点
- 知识图谱可视化页面对短文档显示为空
- 不影响大文档（> 2000 字符）的实体抽取

## 改进方向
1. 规则引擎先行：对短文档加强正则匹配（日期、金额、邮箱等结构化信息）
2. 多文档聚合：同一 KB 的多文档联合抽取实体
3. 阈值调优：降低 LLM 抽取的文本长度下限
